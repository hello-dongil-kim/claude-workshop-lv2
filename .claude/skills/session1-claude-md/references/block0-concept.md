# Block 0: CLAUDE.md 개념

## EXPLAIN

### 실행 환경 — Code 탭 기준

CLAUDE.md를 쓰기 전에, 본인이 쓰는 환경을 한 번 짚고 가자. Claude Code 데스크톱 앱은 **3가지 탭**으로 구성된다:

| 탭 | 용도 | 이번 워크숍 |
| --- | --- | --- |
| **Chat** | 파일 없이 일반 대화 | — |
| **Code** | 폴더를 열고 파일 직접 읽기/수정 | **이 탭에서 진행** |
| **Cowork** | 클라우드 VM에서 백그라운드 독립 작업 | (Session 4에서 다룸) |

> 이 워크숍은 **Code 탭** 또는 IDE/CLI에서 진행한다. 어느 환경이든 CLAUDE.md는 동일하게 작동한다.

### CLAUDE.md란?

Claude Code는 **매 세션마다 기억이 초기화**된다. 어제 나눈 대화, 합의한 규칙, 프로젝트 맥락 — 새 세션을 열면 전부 사라진다.

**CLAUDE.md**는 이 문제를 해결하는 파일이다:
- 프로젝트 루트에 위치한 마크다운 파일
- **매 세션 시작 시 자동으로 읽힘**
- Claude에게 프로젝트의 맥락, 규칙, 작업 방식을 알려주는 "브리핑 문서"

**비유:** 신규 입사자에게 매일 아침 전달하는 온보딩 문서. 매번 같은 설명을 반복할 필요 없이, 문서 하나로 동일한 맥락을 공유한다.

### 핵심 원칙 3가지

**1. 200줄 이하 유지**
- 지시가 많아질수록 뒤쪽 규칙의 준수율이 떨어��는 경향
- 길면 뒤쪽 규칙이 무시될 수 있음
- "이 줄이 없으면 Claude가 실수할까?" 테스트로 불필요한 줄 제거

**2. WHAT → WHY → HOW 구조**

| 순서 | 내용 | 예시 |
|------|------|------|
| WHAT | 프로젝트가 뭔지 | "B2B SaaS 대시보드 기획" |
| WHY | 왜 이걸 하는지 | "고객 이탈률 30% 감소 목표" |
| HOW | 어떻게 작업할지 | "한국어, Markdown, 선택지+추천안" |

**3. 여러분의 강점 활용**
- CLAUDE.md는 코드가 아니라 **마크다운 문서**
- 여러분이 이미 잘 하는 "문서로 설계하기"가 핵심 역량
- 프롬프트 한 줄이 아니라, **구조화된 지시 체계**를 설계하는 것

### CLAUDE.md가 있을 때와 없을 때

| 상황 | CLAUDE.md 없음 | CLAUDE.md 있음 |
|------|----------------|----------------|
| "보고서 써줘" | 영어로 작성, 형식 임의 | 한국어, 존댓말, 지정 포맷 |
| "경쟁사 분석해줘" | 일반적 프레임 사용 | 우리 팀 프레임워크 적용 |
| "파일 정리해줘" | 임의 구조로 저장 | ./outputs/YYYY-MM-DD 형식 |

### 권한 모드 — 4단계 안전장치

> 📌 **변동 정보 (최종 확인: 2026-06-14)** — 아래 모드 이름·단축키·명령어 형식은 바뀔 수 있습니다. 최신값은 공식 문서(https://docs.claude.com)를 확인하세요.

> **🔁 Lv.1 복습 + 확장:** 권한 모드와 컨텍스트 관리는 Lv.1에서 한 번 다룬 주제다. 익숙하면 빠르게 넘어가도 된다. Lv.2에서는 CLAUDE.md와 어떻게 함께 작동하는지에 초점을 맞춘다.

Claude Code는 작업 전에 사용자에게 허락을 구한다. 이 "허락 수준"을 4단계로 조절할 수 있다:

| 모드 | 동작 | 추천 상황 |
|------|------|----------|
| **Plan mode** | 읽기만 가능, 변경 불가 | 탐색·분석, 전략 수립 |
| **Ask permissions** (기본값) | 매 작업마다 허락 요청 | **초보자에게 추천** |
| **Auto accept edits** | 파일 수정은 자동, 명령 실행은 허락 | 익숙해진 후 |
| **Bypass permissions** | 모든 작업 자동 승인 | 비권장 (되돌리기 어려움) |

**전환 방법:** IDE에서는 Claude Code 패널의 모드 선택 드롭다운, 터미널에서는 Shift+Tab으로 전환.

> **초보자 추천:** Ask permissions로 시작하여 Claude의 작업을 하나씩 확인하고, 익숙해지면 Auto accept edits로 전환.

### 컨텍스트 관리 — 성능의 핵심

Claude Code가 한 번에 기억할 수 있는 대화의 총량을 **컨텍스트 윈도우**라고 한다. 책상 위 작업 공간처럼 — 서류가 쌓이면 중요한 문서를 찾기 어려워지듯, 컨텍스트가 차면 Claude의 성능이 떨어진다.

**핵심 관리 명령어:**

| 명령어 | 용도 | 언제 사용하나 |
|--------|------|---------------|
| `/clear` | 대화 내용 완전 초기화 | 새로운 작업 시작 시 |
| `/compact` | 대화를 요약하여 압축 | 대화가 길어졌을 때 |
| `/context` | 컨텍스트 사용량 시각화 | 얼마나 찼는지 확인 |
| `/cost` | 현재 사용량·비용 확인 | 비용 모니터링 |
| `/recap` | 이전 세션 요약 | 작업을 이어서 할 때 |

**실전 관리 전략:**

1. **작업 간 `/clear` 습관화** — 관련 없는 작업 사이에 반드시 `/clear`
2. **수정이 2회 실패하면 리셋** — `/clear` 후 더 명확한 지시로 재시작이 빠름
3. **탐색은 서브에이전트에 위임** — 탐색 과정이 메인 대화를 오염시키지 않음 (Session 4에서 상세히 다룸)

## QUIZ

**문제 1 (회상형):** CLAUDE.md를 200줄 이하로 유지해야 하는 이유는?

| 선택지 | 내용 |
|--------|------|
| A | 길면 뒤쪽 지시가 무시될 수 있어서 |
| B | 컨텍스트 윈도우 용량을 아껴야 해서 |
| C | 읽기 속도가 느려져서 |

**정답:** A — 지시가 많아질수록 뒤쪽 규칙의 준수율이 떨어진다. CLAUDE.md는 "모든 것을 담는 매뉴얼"이 아니라 "핵심만 담는 브리핑"이어야 한다.
- B 오답: 컨텍스트 절약도 부수 효과지만, 핵심 문제는 "뒤쪽 규칙 누락"이다.
- C 오답: 읽기 속도는 사실상 체감되지 않는다 — 준수율 저하가 진짜 위험이다.

**문제 2 (적용형):** 다음 상황에서 가장 적절한 것은? — 보고서를 시킬 때마다 영어로 나오고 형식도 매번 달라서 매번 "한국어로, 표로 정리해줘"를 반복하고 있다.

| 선택지 | 내용 |
|--------|------|
| A | 매번 프롬프트에 "한국어, 표 형식"을 붙여 쓴다 |
| B | CLAUDE.md의 산출물 규칙에 "언어: 한국어 / 기본 포맷: 표 포함 마크다운"을 한 번 적어둔다 |
| C | 보고서가 나온 뒤 매번 직접 번역하고 표로 다시 짠다 |

**정답:** B — 매 세션 반복되는 규칙은 CLAUDE.md에 한 번 적어두면 이후 자동 적용된다. 반복 지시·수동 작업이 사라진다.
- A 오답: 작동은 하지만 매번 입력하는 소모가 그대로 남는다 — 반복 규칙은 문서화가 정답.
- C 오답: 가장 비효율적. 원인(규칙 미설정)을 두고 결과만 수습하는 방식이다.
