# Block 0: Skills 개념

## EXPLAIN

### Skills란?

Claude Code에서 지식과 맥락을 주입하는 방법은 크게 2가지다:

| 구분 | 일반 프롬프트 | CLAUDE.md | Skills |
|------|-------------|-----------|--------|
| **비유** | 매번 구두로 지시하기 | 사무실 벽에 붙인 사규 | 업무별 매뉴얼 |
| **적용 범위** | 그 대화에서만 | 모든 대화에 항상 적용 | 관련 상황에서만 자동 로딩 |
| **호출 방식** | 직접 입력 | 자동 (항상 배경에) | 자동 + `/스킬명` 수동 호출 |
| **적합한 용도** | 일회성 질문 | 프로젝트 공통 규칙, 톤, 포맷 | 반복되는 특정 업무의 전문 지식 |

**CLAUDE.md**는 매 세션 시작 시 무조건 읽힌다. 항상 적용되는 규칙이 들어간다.

**Skills**는 대화 내용을 보고 Claude가 **알아서 로딩**한다. 사용자가 "인터뷰 스크립트 작성해줘"라고 말하면, UX 리서치 Skill이 자동으로 활성화되는 식이다.

> **참고:** Session 1에서 배운 프로젝트 구조를 떠올려보자. `.claude/` 폴더에는 commands/(수동 호출), rules/(자동 로딩 규칙), skills/(자동 트리거), agents/(독립 위임) 4개 확장 폴더가 있었다. Commands는 `/명령어`로 수동 호출하는 단일 프롬프트 파일이고, Skills는 대화 맥락에 따라 자동 트리거되는 상위 호환이다. 이 세션에서는 Skills에 집중한다.

### 내장 스킬 — 바로 쓸 수 있는 도구

Claude Code에는 미리 만들어진 내장 스킬이 있다. 별도 설치 없이 바로 사용할 수 있다:

| 내장 스킬 | 용도 | 사용 예시 |
|-----------|------|----------|
| `/batch` | 대량 파일 작업을 병렬 처리 | "input/의 모든 CSV를 각각 요약해줘" |
| `/simplify` | 생성된 결과물을 여러 에이전트가 품질 검토·개선 | 코드/문서 품질 향상 |
| `/loop 시간 /명령` | 주기적 반복 실행 | `/loop 5m /competitor-check` |

이 내장 스킬들은 뒤에서 배울 Subagent(Session 4)를 활용한다. 지금은 "이런 것이 있다"는 정도만 알아두면 된다.

### 전문 컨설턴트 팀 비유

회사에 전문 컨설턴트 팀이 대기하고 있다고 상상하자:

| 컨설턴트 | 전문 분야 | 호출 조건 |
|----------|-----------|-----------|
| UX 리서치 전문가 | 인터뷰, 사용성 테스트, 설문 | "리서치" 관련 대화가 나오면 합류 |
| 서비스 기획 전문가 | PRD, 화면 설계, 정책 정의 | "기획" 관련 대화가 나오면 합류 |
| 데이터 분석가 | SQL, 코호트, A/B 테스트 | "데이터" 관련 대화가 나오면 합류 |

Skills는 이 컨설턴트들과 같다. **평소에는 대기하고, 맥락이 맞으면 자동으로 투입**된다.

### CLAUDE.md vs Skills 상세 비교

| 비교 항목 | CLAUDE.md | Skills |
|-----------|-----------|--------|
| **로딩 시점** | 매 세션 시작 시 항상 | 대화 맥락에 따라 자동 |
| **파일 구조** | 단일 `.md` 파일 | `SKILL.md` + `references/` + `examples/` |
| **용량 영향** | 항상 컨텍스트 차지 | 필요할 때만 로딩 (효율적) |
| **적합한 상황** | 프로젝트 공통 규칙, 톤, 포맷 | 특정 도메인의 전문 지식과 워크플로우 |
| **수동 호출** | 불가 (항상 자동) | `/스킬명`으로 수동 호출도 가능 |

### 자동 트리거는 어떻게 작동하나?

1. 사용자가 메시지를 입력한다
2. Claude Code가 설치된 모든 Skill의 **description** 필드를 스캔한다
3. 메시지의 키워드/맥락이 description과 매칭되면 **해당 Skill을 자동 로딩**
4. Skill의 SKILL.md 내용이 Claude의 컨텍스트에 추가된다

핵심: **description 필드가 트리거 조건을 결정**한다. description을 어떻게 쓰느냐에 따라 Skill이 로딩되는 시점이 달라진다.

### 이 작업, Skill로 만들어야 할까? — 판단 체크리스트

아래 4가지 조건 중 **2개 이상 해당하면** Skill로 만들 가치가 있다:

| 조건 | 질문 | 예시 |
|------|------|------|
| **반복성** | 주 1회 이상 같은 유형의 작업을 하는가? | 매주 보고서, 매일 회의록 정리 |
| **규칙성** | 이 작업에 우리만의 규칙이 있는가? | 브랜드 톤, 특정 포맷, 체크리스트 |
| **일관성** | 결과물 품질이 들쑥날쑥하면 안 되는가? | 고객 응대, 공식 문서, 보고서 |
| **공유성** | 다른 사람도 같은 기준으로 해야 하는가? | 팀원 인수인계, 외주 전달 기준 |

> **프롬프트를 매번 쓰는 건 소모이고, Skill로 만들어두면 자산이 된다.**

### 언제 무엇을 쓸까? — 의사결정 가이드

"이 규칙을 CLAUDE.md에 넣을까, Skill로 만들까, Command로 만들까?"

| 조건 | 추천 | 이유 |
| ---- | ---- | ---- |
| 모든 세션에서 항상 적용 | **CLAUDE.md** (또는 rules/) | 항상 자동 로딩. 산출물 포맷, 언어, 톤 등 |
| 특정 주제일 때만 필요 | **Skill** | 키워드 기반 자동 트리거. UX 리서치, PRD 등 |
| 내가 원할 때만 실행 | **Command** | `/명령어`로 수동 호출. 주간 보고, 경쟁사 스캔 등 |
| 독립적인 보조 작업 | **Subagent** | 별도 컨텍스트에서 실행. 리서치, 데이터 정리 등 |

**간단 판단법:**
- "매번 적용해야 해?" → CLAUDE.md
- "주제에 따라 자동으로?" → Skill
- "내가 부를 때만?" → Command
- "따로 시켜서 결과만 받을래?" → Subagent

### 자주 하는 실수 — 안티패턴 5가지

| 안티패턴 | 왜 문제인가 | 올바른 방법 |
|----------|-------------|-------------|
| CLAUDE.md에 500줄 넣기 | 뒤쪽 규칙이 무시됨 | 200줄 이하 유지, 나머지는 rules/ 또는 Skill로 분리 |
| 모든 지식을 Skill로 만들기 | 항상 적용할 규칙까지 Skill로 만들면 트리거 안 될 때 누락 | 항상 적용 → CLAUDE.md, 주제별 → Skill |
| description에 추상적 설명 | "도와줍니다" 같은 표현은 트리거 키워드가 아님 | 구체적 키워드 나열: "인터뷰, 설문, FGI 키워드 시 트리거" |
| 단순 질문에 Subagent 호출 | 불필요한 컨텍스트 전환과 토큰 소비 | Subagent는 중간 과정이 많은 보조 작업에만 사용 |
| 모든 작업에 Agent Teams 구성 | 토큰 비용 급증, 복잡성 증가 | 단일 Subagent로 충분한지 먼저 판단 |

> ✅ **이렇게 되면 성공:** 스킬 이름을 직접 부르지 않았는데도, 대화 주제와 맞는 스킬이 자동으로 로딩되어 그 역할·규칙이 반영된 응답이 나오면 성공.

## QUIZ

**문제 1 (회상형):** Skills가 Commands와 다른 가장 핵심적인 차이는?

| 선택지 | 내용 |
|--------|------|
| A | Skills는 파일이 더 많다 (SKILL.md + references/) |
| B | Skills는 대화 맥락에 따라 자동으로 로딩된다 |
| C | Skills는 인자를 전달할 수 없다 |

**정답:** B — Commands는 사용자가 `/명령어`를 직접 입력해야 실행되지만, Skills는 대화 내용의 키워드/맥락을 보고 Claude가 자동으로 로딩한다. 이것이 "전문 컨설턴트 팀"에 비유되는 이유다.
- A 오답: 파일 개수는 부수적 차이일 뿐, 핵심은 "호출 방식"이다.
- C 오답: 인자 전달은 Command의 특징이다 — Skills와의 본질적 차이가 아니다.

**문제 2 (적용형):** 다음 상황에서 가장 적절한 것은? — 모든 문서를 항상 "한국어, 존댓말, 마크다운"으로 받고 싶다. 주제와 무관하게 늘 적용돼야 한다.

| 선택지 | 내용 |
|--------|------|
| A | 이 규칙을 Skill로 만들어 description에 키워드를 넣는다 |
| B | 이 규칙을 CLAUDE.md(또는 rules/)에 넣는다 |
| C | 이 규칙을 Command로 만들어 매번 `/format`을 호출한다 |

**정답:** B — "주제와 무관하게 항상 적용"은 자동 로딩되는 CLAUDE.md(또는 rules/)의 영역이다.
- A 오답: Skill은 키워드가 맞을 때만 트리거되므로, 키워드가 없는 대화에선 규칙이 누락된다.
- C 오답: Command는 내가 부를 때만 실행 — "항상"이라는 요건과 맞지 않는다.
