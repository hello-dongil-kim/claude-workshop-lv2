# Block 2: 나만의 첫 Skill 만들기

## EXPLAIN

### 기획자용 Skills 후보 5개

기획자, UX 리서처, PM이 가장 많이 쓸 Skill 후보는 다음과 같다:

| Skill | 폴더명 | 자동 트리거 키워드 | 활용 장면 |
|-------|--------|-------------------|-----------|
| UX 리서치 | `ux-research` | 인터뷰, 사용성 테스트, 설문, FGI, 리서치 설계 | 리서치 플랜 수립, 인터뷰 스크립트 작성 |
| 서비스 기획 | `service-planning` | PRD, 화면 설계, 정책 정의, 기능 명세, 사용자 스토리 | PRD 초안, 화면 흐름 정의 |
| 데이터 분석 | `data-analysis` | 지표 분석, 코호트, A/B 테스트, 퍼널, 리텐션 | 지표 리뷰, 실험 설계 |
| 워크숍 설계 | `workshop-design` | 워크숍, 퍼실리테이션, 아이데이션, 디자인 스프린트 | 워크숍 기획, 퍼실리테이션 가이드 |
| 이해관계자 커뮤니케이션 | `stakeholder-comms` | 보고서, 주간 보고, 경영진 보고, 상태 업데이트 | 보고 문서, 이해관계자 맞춤 메시지 |

### Skill 요청 5요소 — Claude에게 잘 시키는 법

Claude에게 "Skill 만들어줘"라고 말할 때, 아래 5가지를 담으면 퀄리티가 확 올라간다:

| 요소 | 설명 | 예시 |
|------|------|------|
| **작업 정의** | 이 Skill이 할 일 | "인터뷰 결과를 정리하는 스킬" |
| **트리거 상황** | 언제 자동으로 작동할지 | "인터뷰, 사용자 리서치, FGI 키워드 시" |
| **규칙/가이드** | 따라야 할 규칙들 | "발견별 근거 표기, 출처 필수" |
| **톤앤매너** | 결과물의 말투/분위기 | "분석적이되 읽기 쉽게" |
| **결과물 형태** | 어떻게 생긴 결과를 원하는지 | "핵심 발견 → 상세 분석 → 출처 순서" |

> **Lv.1에서 배운 ODAF(Outcome-Data-Action-Format)와 비슷하다.** ODAF가 "좋은 프롬프트"의 4요소였듯, 이 5요소는 "좋은 Skill 요청"의 구조다.

### Before/After — Skill 요청 품질 비교

같은 Skill을 요청해도 어떻게 말하느냐에 따라 결과가 다르다:

**Bad:** `"회의록 정리 스킬 만들어줘"`
→ 일반적인 회의록 가이드가 나온다. 내 팀의 포맷, 톤, 필수 항목을 모름.

**Good:** `"우리 팀 회의록 정리 스킬 만들어줘. 녹취록을 주면 이렇게 정리해줘: 참석자, 핵심 안건 3~5개, 결정사항, 액션아이템(담당자+기한 테이블), 미결 사항. 톤은 간결하고 실무적으로. 발언자 이름은 빼고 내용 중심으로."`
→ 내 팀에 딱 맞는 회의록 Skill이 나온다.

### Skill 만들기 3단계

**Step 1: 폴더 생성**
```
.claude/skills/{skill-name}/
├── SKILL.md
├── references/          ← 보충 자료 (선택)
└── examples/            ← 예시 파일 (선택, 없어도 됨)
```

**Step 2: SKILL.md 작성** (Claude에게 시키면 됨)
- frontmatter에 `name`과 `description` 작성
- description에 **구체적 트리거 키워드** 나열
- 본문에 역할, 원칙, 참조 파일 정의

**Step 3: 자동 트리거 테스트**
- 새 세션을 열거나 대화를 시작
- description에 적은 키워드를 포함한 질문을 던짐
- Claude가 해당 Skill을 자동 로딩하는지 확인

### Skill 품질 체크리스트 — 만든 후 반드시 확인

| # | 체크 항목 | 확인 방법 |
|---|-----------|-----------|
| 1 | **description에 트리거 키워드가 있는가?** | `"이 스킬의 description 보여줘"` → 실제로 쓸 키워드 포함 확인 |
| 2 | **예시가 포함되어 있는가?** | SKILL.md 또는 examples/에 좋은 산출물 샘플이 있는지 |
| 3 | **하나의 Skill이 하나의 역할만 하는가?** | 만능 Skill(블로그+인스타+이메일)은 결과가 애매해짐. 분리할 것 |
| 4 | **실제로 테스트해봤는가?** | 자동 호출 + `/스킬명` 수동 호출 + "어떤 스킬 있어?" 인식 확인 |
| 5 | **Skill이 너무 길지 않은가?** | SKILL.md 500줄 이하. 길면 references/로 분리 |

### 템플릿 활용

@templates/skill-template.md에 기획자용 SKILL.md 뼈대 템플릿이 준비되어 있다. 이 템플릿을 복사해서 자신의 Skill에 맞게 수정하면 빠르게 시작할 수 있다.

**Step 1: 후보 선택**
- 위 5개 후보 중 지금 가장 필요한 Skill 하나를 고르세요
- 추천: 평소 업무에서 가장 자주 하는 작업과 연결된 Skill

**Step 2: Claude에게 Skill 만들어달라고 요청 (5요소 활용)**

```text
{내가 고른 skill-name} Skill을 만들어줘.
@templates/skill-template.md 를 참고해서 .claude/skills/ 에 생성해줘.

작업 정의: {이 Skill이 할 일}
트리거 상황: {키워드 3~5개 나열}
규칙: {핵심 규칙 2~3개}
톤: {결과물의 말투}
결과물 형태: {산출물 구조}
```

- Claude가 생성한 SKILL.md를 검토
- description의 키워드가 내가 실제로 쓸 표현인지 확인
- 마음에 안 드는 부분만 수정 지시
- **핵심: 직접 YAML을 작성할 필요 없다. Claude에게 시키고 결과를 검토하면 된다**

> **참고:** Claude 데스크톱 앱에는 **Skill Creator**가 내장되어 있어, 대화하듯이 Skill을 만들 수 있다. Claude Code에서는 위처럼 자연어로 요청하면 동일한 결과를 얻는다.

**Step 3: 자동 트리거 테스트**
- 새 대화를 시작하고, description에 적은 키워드를 포함해서 질문
- 예: `"사용성 테스트 계획을 세워줘"` (ux-research를 만들었다면)
- Claude가 Skill의 역할/원칙을 반영한 응답을 하는지 확인

## QUIZ

**문제:** Skill 폴더 구조에서 references/의 역할은?

| 선택지 | 내용 |
|--------|------|
| A | SKILL.md의 백업 복사본을 보관하는 곳 |
| B | Skill이 실행될 때 사용자에게 자동 출력되는 파일 |
| C | SKILL.md가 참조하는 보충 자료를 담는 곳 |

**정답:** C — references/는 Skill이 더 전문적인 산출물을 만들기 위해 참조하는 보충 자료 폴더다. SKILL.md 본문에서 `@references/파일명.md`로 연결하면 Claude가 함께 읽고 활용한다. 필수는 아니지만, 있으면 산출물 품질이 올라간다.
