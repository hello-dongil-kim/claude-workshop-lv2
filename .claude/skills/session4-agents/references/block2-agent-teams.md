# Block 2: Agent Teams

## EXPLAIN

### Agent Teams란?

Block 0~1에서 배운 Subagent는 **수직 구조**다. 메인 Claude가 지시하고, Subagent가 수행하고, 결과를 보고한다. 한 명의 보스와 여러 명의 보조원.

**Agent Teams**는 다른 접근이다:
- 여러 전문가 에이전트가 **수평적으로** 소통하며 병렬 협업
- 각 에이전트는 고유한 전문 분야를 가지고, 서로의 결과를 참조
- Lead Agent가 전체를 조율하되, 각 에이전트는 자율적으로 판단

### Subagent(수직) vs Agent Teams(수평) 비교

| 항목 | Subagent (수직) | Agent Teams (수평) |
|------|-----------------|-------------------|
| **구조** | 메인 → Subagent (상하 관계) | Lead + 전문가들 (협업 관계) |
| **소통 방향** | 단방향 (지시 → 결과) | 양방향 (에이전트 간 참조) |
| **작업 방식** | 순차적 위임 | 병렬 협업 |
| **적합한 상황** | 단일 보조 작업 (리서치, 정리) | 복합 프로젝트 (다관점 분석) |
| **토큰 소비** | 낮음~중간 | 높음 |
| **복잡도** | 낮음 | 높음 |
| **안정성** | 안정적 | 실험적 |

### 기획 프로젝트 시나리오

새 서비스 기획 프로젝트를 Agent Teams으로 구성한다면:

```
                    ┌─────────────────┐
                    │   Lead Agent    │
                    │  (서비스 기획)   │
                    └────────┬────────┘
                             │ 조율
            ┌────────┬───────┼───────┬────────┐
            │        │       │       │        │
      ┌─────┴──┐ ┌───┴───┐ ┌┴─────┐ ┌┴──────┐
      │Agent A │ │Agent B│ │Agent C│ │Agent D│
      │시장    │ │사용자 │ │기술   │ │비즈니스│
      │리서치  │ │리서치 │ │타당성 │ │모델   │
      └────────┘ └───────┘ └──────┘ └───────┘
           │          │        │         │
           ▼          ▼        ▼         ▼
      시장 규모    페르소나   기술 제약  수익 모델
      경쟁 분석    여정 맵    구현 난이도 비용 구조
      트렌드      니즈 분석   대안 검토  BEP 분석
```

**흐름:**
1. Lead Agent가 프로젝트 목표와 범위를 정의
2. Agent A~D가 각자 전문 분야를 병렬로 리서치
3. 각 에이전트의 결과를 Lead Agent가 취합하여 종합 보고서 작성
4. 에이전트 간 교차 참조 — 예: Agent C(기술)가 Agent A(시장)의 결과를 보고 기술 타당성 평가 조정

### Agent Teams 구현 예시

실제로 Agent Teams를 설정한다면, Lead Agent와 Specialist Agent를 각각 `.claude/agents/`에 배치한다:

**Lead Agent — 서비스 기획 총괄:**

```yaml
---
name: service-lead
description: |
  서비스 기획 총괄 에이전트. 새 서비스 분석, 종합 보고, 기획 총괄 키워드에 트리거.
  시장/사용자/기술 분석 결과를 취합하여 종합 판단을 내린다.
model: opus
maxTurns: 25
tools:
  - Read
  - Write
  - Agent
---

## 역할
서비스 기획 프로젝트의 총괄 에이전트. 각 전문 에이전트에게 분석을 위임하고 결과를 취합한다.

## 워크플로우
1. 프로젝트 목표와 분석 범위를 정의
2. @market-researcher에게 시장 분석 위임
3. @user-researcher에게 사용자 분석 위임
4. 각 결과를 취합하여 종합 보고서 작성

## 산출물 형식
### [프로젝트명] 종합 분석 보고서
- **시장 분석 요약**: (market-researcher 결과)
- **사용자 분석 요약**: (user-researcher 결과)
- **종합 판단**: Go/No-Go 및 근거
- **다음 단계**: 액션 아이템
```

**Specialist Agent — 시장 리서치:**

```yaml
---
name: market-researcher
description: |
  시장 리서치 에이전트. 시장 규모, 경쟁사, 트렌드 분석 키워드에 트리거.
model: sonnet
maxTurns: 15
tools:
  - WebSearch
  - Read
  - Write
---

## 역할
시장 조사 전문 에이전트. 시장 규모, 경쟁 환경, 트렌드를 분석한다.

## 원칙
- 사실(Fact)과 추정(Estimate)을 명확히 구분
- 출처를 반드시 표기
- 핵심 발견 3~5개를 먼저 제시
```

**핵심 포인트:**
- Lead Agent는 `model: opus` + `tools: Agent`로 다른 에이전트를 호출할 수 있다
- Specialist는 `model: sonnet`으로 비용 효율적으로 운영
- 각 에이전트는 독립 컨텍스트에서 작업하므로, 서로의 중간 과정이 오염되지 않는다

### 주의사항

Agent Teams는 강력하지만, 반드시 알아야 할 제약이 있다:

| 항목 | 내용 |
|------|------|
| **토큰 소비** | 에이전트 수 x 각 에이전트의 작업량. 비용이 빠르게 증가 |
| **복잡성** | 에이전트 간 소통 설계가 어려움. 결과가 충돌할 수 있음 |
| **안정성** | 실험적 기능. 예상치 못한 동작이 발생할 수 있음 |
| **적용 기준** | 단순 작업은 Subagent로 충분. Teams는 다관점 분석이 필수인 프로젝트에만 사용 |

**실전 판단 기준:**

| 상황 | 추천 |
|------|------|
| "이 주제 리서치해줘" | Subagent 1개 |
| "회의록 정리해줘" | Subagent 1개 |
| "새 서비스의 시장성, 사용자 니즈, 기술 타당성, 수익 모델을 종합 분석해줘" | Agent Teams |
| "경쟁사 3곳을 비교 분석해줘" | Subagent 1개 (또는 병렬 3개) |

### Agent Teams의 대안 — `/batch` 내장 스킬

Agent Teams가 "서로 다른 전문가들의 협업"이라면, `/batch`는 **"같은 작업을 여러 입력에 병렬 적용"**하는 다른 접근이다. Session 2 block0에서 소개한 내장 스킬 중 하나로, 파일 수만큼 에이전트를 자동 생성하여 동시에 작업한다.

```text
/batch competitor-list.csv 의 30개 회사를 각각 1줄로 요약해줘
```

- 30개 파일을 순차 처리하면 30분 → 병렬 처리하면 3~5분
- 각 에이전트는 독립 컨텍스트에서 작업하므로 메인 대화는 깨끗
- Agent Teams보다 단순한 구조 — "같은 작업을 여러 입력에 반복"할 때 적합

### 3가지 도구 — 어떻게 골라 쓰나

| 도구 | 적합한 상황 | 예시 |
| ---- | ---------- | ---- |
| **Subagent (1개)** | 단일 보조 작업 위임 | "이 주제 리서치해줘" |
| **`/batch`** | 같은 작업을 여러 항목에 병렬 적용 | "30개 경쟁사 각각 1줄 요약" |
| **Agent Teams** | 서로 다른 전문 분야가 협업하는 복합 분석 | "시장·사용자·기술·BM 종합 분석" |

> **선택 기준:** 작업 종류가 1개면 Subagent 또는 `/batch`(여러 입력일 때), 작업 종류가 여러 개면 Agent Teams.

## QUIZ

**문제:** 다음 중 Agent Teams가 가장 적합한 상황은?

| 선택지 | 내용 |
|--------|------|
| A | 경쟁사 A의 최신 소식을 리서치해야 할 때 |
| B | 회의록을 정리하고 액션 아이템을 추출해야 할 때 |
| C | 새 서비스의 시장성, 사용자 니즈, 기술 타당성을 동시에 분석해야 할 때 |

**정답:** C — Agent Teams는 여러 전문 분야의 관점이 동시에 필요한 복합 프로젝트에 적합하다. A와 B는 단일 Subagent로 충분히 처리할 수 있는 작업이다.
